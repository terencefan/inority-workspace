# Slides Playbook

适用于主路径围绕演示文稿、汇报 deck、培训课件、品牌 H5 或 handbook 可内嵌 slides 项目产出的任务。

不要把普通 spec、README、总结文档或运维 runbook 误判成 slides 任务。

## 何时使用

- 目标主要是交付一份 slides / PPT / H5 演示站点
- 关键风险来自受众定位、叙事顺序、素材口径、图表证据、品牌模板或导出形态不一致
- 执行路径的核心是“现在有哪些素材、要讲给谁、如何收敛成可交付 deck”

## 规划重点

- 侦察阶段的目标是准备素材：补齐受众、场景、来源素材、模板口径、图源状态和交付格式
- slides 的 QA 要比普通文档更密；不能只确认总主题，还要逐层确认 section 主线和 slide 目标
- 最终产物默认不是 Markdown slide deck，也不是 office 中间稿；它应是一套可独立打开、可静态托管、可被 handbook 内嵌的独立 slides 项目
- 如果 slides 里包含架构图、线框图、关系图或页面流转图，默认统一收敛为 inline `SVG`
- `DOT`、白板导图、设计稿或其他图源格式可以作为上游素材存在，但进入 slides 编排时必须落成可控的最终 `SVG`
- 每一页默认都要同时具备三类内容：`SVG` 线框图、交互说明、素材清单

## 默认技术路线

- 默认技术路线是 `Vite + GSAP + Lenis`
- 默认交付目录是一个独立静态项目目录，入口固定为 `index.html`
- 默认运行形态是“品牌叙事型 slides 站点”，而不是 React 应用壳、Markdown deck 或 office 导出中间态
- 默认支持：
  - 鼠标滚动 / 触控板滚动推进
  - 页内 `ScrollTrigger + timeline` 流转
  - 页间导航、跳页、可内嵌到 handbook
  - 构建后产出可直接静态托管的 `dist/`

## 技术选型决策路径

- 第一优先级是品牌表达和展示控制力，不是通用文档复用率
- 如果目标是品牌型、发布会型、叙事型 slides，优先选择 `Vite + GSAP + Lenis`
- 选择 `Vite`，因为它天然以 `index.html` 为入口，构建产物适合静态托管，也适合作为 handbook 的嵌入资源
- 选择 `GSAP + ScrollTrigger`，因为页内流转核心是 timeline 编排、scroll 驱动和 pin/scrub 能力
- 选择 `Lenis`，因为它负责统一滚轮 / 触控板的滚动手感，并让页间推进和页内流转处于同一条滚动运行面
- 不默认选择 React；只有当 slides 本身已经退化成复杂应用界面、状态管理和组件复用远高于展示编排时，才考虑升级
- 不默认选择 `Slidev` / `Marp`；它们偏 Markdown-first，适合内容驱动，不适合“强品牌表达 + 高自由度动效”
- 不默认选择 `reveal.js`；它更适合标准 presentation framework，不如自建 `Vite + GSAP + Lenis` shell 自由

## 最终产物约束

- 每份 slides authority 最终都应能落成一个独立目录，例如 `slides/<topic>/`
- 目录内必须有可直接打开的入口 `index.html`
- 如果项目走构建链路，最终必须能产出 `dist/index.html`
- slides 项目必须既能脱离 handbook 独立打开，也能作为 handbook 的嵌入内容被加载
- 如果某份 slides 规划最终只能导出成普通 Markdown、PDF 或 office 文件，而没有对应静态项目落点，默认视为未收敛完成

## 常见证据

- 目标受众、汇报场景、时间预算、讲述重点
- 用户对每个 section 主线的明确确认
- 用户对每个 slide 页目标的明确确认
- 来源文档、数据表、截图、图表、参考 deck、品牌模板
- 架构图源、线框草图、页面流转草图、导出的 `SVG` 成品和其命名约定
- 每页交互说明、页面状态切换、热点区域、动效触发条件
- 每页素材清单：标题文案、正文文案、配图、图标、数据、引用来源
- 页数上限、导出格式、演讲备注、交付截止时间

## 写作约束

- 重点是把现有素材转化为可交付 deck 的执行路径，不要退化成普通 spec 或 README
- 改用大纲式层级：二级标题是大纲视图，三级标题是 section，四级标题是 slide
- `大纲视图` 开头必须先放一张思维脑图，用来总览整套 section / slide 结构
- 不能只让用户确认一次“大纲差不多”；每个 `### section` 的主线都要有单独确认，每个 `#### slide` 的页目标也都要有单独确认
- 大纲和详细 slide 顺序必须严格一一对应
- 每一页的 planning 内容默认至少写清：
  - GitHub 风格的目标 Note
  - 嵌入式 `SVG` 线框图
  - 交互说明
  - 素材清单
  - 页级验收标准
- 对品牌型 slides，`交互说明` 不应只写“下一页 / 上一页”；应显式写清页内流转、滚动推进、导航方式和最终嵌入 / 独立打开形态

## 页级默认结构

每个 `#### slide` 默认应包含：

1. GitHub 风格的目标 Note
2. 嵌入式 `SVG` 线框图
3. 交互说明
4. 素材清单
5. 验收标准

其中：

- `目标 Note`：必须写成 `> [!NOTE]`，块内至少包含 `页目标` 和 `QA 链接`
- `QA 链接`：必须可点击，直接回指确认该页目标的真实问答
- `SVG` 线框图：必须直接以内联 `<svg>...</svg>` 形式嵌在该页正文里
- `交互说明`：描述翻页、点击、悬停、滚动、展开、切换或动效触发逻辑
- `素材清单`：列出该页实际需要的文案、图片、图标、数据、图表或引用来源

## 章节层级约定

- `##`：大纲视图
- `###`：section
- `####`：slide

在 `## 大纲视图` 内，默认顺序是：

1. `## 思维脑图`
2. `### section`
3. `#### slide`

## QA 收敛要求

- 至少要有 `1` 条问答确认整套 slides 的总目标 / 受众 / 交付形态
- 每个 `### section` 至少要有 `1` 条问答确认该 section 的主线、作用和边界
- 每个 `#### slide` 至少要有 `1` 条问答确认该页页目标、是否保留、以及是否需要调整顺序
- 每个 `#### slide` 的目标 Note 都必须带一个能点回这条问答的 QA 链接

## 大纲与详细映射规则

- `## 大纲视图` 中出现的每个 `#### slide`，都必须在详细页列表里找到唯一对应项
- 详细页列表不允许出现大纲里没有的 slide
- 如果大纲顺序是 `A -> B -> C`，详细页顺序也必须是 `A -> B -> C`
- 调整页顺序时，必须同时更新脑图、大纲和详细页内容，不能只改其中一边
- 大纲视图里的每个 `#### slide` 都必须直接包含嵌入式 `SVG` 线框图，不允许只保留文字占位
