# Slides Playbook

适用于主路径围绕演示文稿、汇报 deck、培训课件、品牌 H5 或 handbook 可内嵌 slides 项目产出的任务。

不要把普通 spec、README、总结文档或运维 runbook 误判成 slides 任务。

## 何时使用

- 目标主要是交付一份 slides / PPT / H5 演示站点
- 关键风险来自受众定位、叙事顺序、素材口径、图表证据、品牌模板或导出形态不一致
- 执行路径的核心是“现在有哪些素材、要讲给谁、如何收敛成可交付 deck”
- 默认模板口径统一到 skill 内部的 `assets/demo/`，不要再把 `brand-fancy` 当成新的权威名字继续扩散
- `demo` 系模板默认按占满 `16:9` 屏幕来设计，这是硬约束，不是视觉建议
- 如果最终实现对齐 `demo`，图表、架构图和信息图的配色也要继承 `demo` 的主题色系，不要各页各自发明一套新颜色

## 规划重点

- 侦察阶段的目标是准备素材：补齐受众、场景、来源素材、模板口径、图源状态和交付格式
- slides 的 QA 要比普通文档更密；不能只确认总主题，还要逐层确认 section 主线和 slide 目标
- 每个 `section` 默认必须先有一张章节标题页，然后才能进入该 section 的内容页
- 整套 deck 默认必须以一张 `致谢` 页收尾
- 最终产物默认不是 Markdown slide deck，也不是 office 中间稿；它应是一套可独立打开、可静态托管、可被 handbook 内嵌的独立 slides 项目
- 如果 slides 里包含架构图、线框图、关系图或页面流转图，默认统一收敛为 inline `SVG`
- `DOT`、白板导图、设计稿或其他图源格式可以作为上游素材存在，但进入 slides 编排时必须落成可控的最终 `SVG`
- 对架构图页，允许先产出 `DOT` 再渲染成最终 `SVG`；但 planning 冻结和 review 的对象始终是最终嵌入 slide 的那张 `SVG`
- 每一页默认都要同时具备三类内容：inline `SVG` 线框图、交互说明、精确素材清单
- `SVG` 灯箱预览不是线框图放大图；它应指向一个可在 handbook 项目里直接访问的预览页或预览资源
- 画 planning 线框图时，默认使用直角矩形，不使用圆角矩形；除非用户明确要求保留圆角
- 重要补充：inline `SVG` 线框图负责描述页面结构；如果某一页本身需要“架构图”作为主视觉，那么 planning 阶段必须额外给出可 review 的完整彩色 `SVG` 架构图，不能只停留在线框占位
- 也就是说，架构图页至少要有两层冻结：页面结构线框，以及实际架构图成品的彩色 `SVG` review 目标
- 一旦该彩色架构图已经作为独立嵌入资产存在，线框图就只需要留出这张图在版面中的位置，不应再在线框层重复绘制架构内容
- 这里的“架构图”不是普通流程图。它必须能表达真正的关联关系、边界约束、authority 分工、数据流、分布式落点、故障域或分层结构，而不是只画一个 `A -> B -> C` 的讲解路径
- 如果像 canary 这类主题已经在现行 spec 中有成熟的架构 / 网络 / 集群 / observability 图法，那么 slides 里的嵌入架构图应优先继承这些 spec 语义，而不是退化成更轻的 PPT 流程图
- 不管是线框图还是嵌入式架构图，文字都不允许超出边界；一旦出现溢出，必须通过改文案、拆行、缩字号或扩版位来解决
- 语义继承 spec，不等于视觉也照抄 spec；如果 deck 落在 `demo` 模板上，配色、线条明度、底色对比和强调色应回到 `demo` 的主题变量体系内统一
- 默认要主动避免接线重叠；如果页面采用分层架构图，优先使用“层间左到右、层内自上到下”的排布，只有在 spec 语义更强时才打破这个默认

## 默认技术路线

- 默认技术路线是 `Vite + GSAP + Lenis`
- 默认实现模板是 skill 内部的 `assets/demo/`
- 默认交付目录是一个独立静态项目目录，入口固定为 `index.html`
- 默认运行形态是“品牌叙事型 slides 站点”，而不是 React 应用壳、Markdown deck 或 office 导出中间态
- 默认主舞台比例固定按 `16:9` 满屏组织；页面内容、字号和模块密度都必须服从这个舞台约束
- 默认支持：
  - 鼠标滚动 / 触控板滚动推进
  - 页内 `ScrollTrigger + timeline` 流转
  - 页间导航、跳页、可内嵌到 handbook
  - 构建后产出可直接静态托管的 `dist/`

## 本地展示逻辑分类

规划态不要把每一页都当成“全新内容类型”。优先用展示逻辑来分类，并在每页明确标注 `当前 slide 类型：<展示逻辑分类>/<变体名>`。

- `display-logic/three-panel`
  适合三段式、三列并列、三对象对照页
- `display-logic/comparison`
  适合左右或上下对比、边界卡、正反面对照页
- `display-logic/linear-flow`
  适合线性链路、请求流、依赖流、步骤串流页
- `display-logic/waterfall`
  适合瀑布流、逐层下钻、从总览到细节的连续展开页
- `display-logic/timeline`
  适合时间线、阶段推进、进入到演进的顺序叙事页
- `display-logic/dashboard-grid`
  适合多卡片看板、指标拼板、证据聚合页
- `display-logic/annotation-board`
  适合规则、权限、配置、注释密集型说明页
- `display-logic/hero-rail`
  适合主视觉定调 + 辅助侧栏 / 侧轨说明页
- `display-logic/hero-detail`
  适合单主视觉 + 右侧或下方细节说明页

如果现有展示逻辑不够，再新增变体名；但不要先按内容主题发明类型名。

## 技术选型决策路径

- 第一优先级是品牌表达和展示控制力，不是通用文档复用率
- 如果目标是品牌型、发布会型、叙事型 slides，优先选择 `Vite + GSAP + Lenis`
- 选择 `Vite`，因为它天然以 `index.html` 为入口，构建产物适合静态托管，也适合作为 handbook 的嵌入资源
- 选择 `GSAP + ScrollTrigger`，因为页内流转核心是 timeline 编排、scroll 驱动和 pin/scrub 能力
- 选择 `Lenis`，因为它负责统一滚轮 / 触控板的滚动手感，并让页间推进和页内流转处于同一条滚动运行面
- 不默认选择 React；只有当 slides 本身已经退化成复杂应用界面、状态管理和组件复用远高于展示编排时，才考虑升级
- 不默认选择 `Slidev` / `Marp`；它们偏 Markdown-first，适合内容驱动，不适合“强品牌表达 + 高自由度动效”
- 不默认选择 `reveal.js`；它更适合标准 presentation framework，不如自建 `Vite + GSAP + Lenis` shell 自由

## 最终产物约束

- 每份 slides authority 最终都应能落成一个独立目录，例如 `slides/<topic>/`
- 但在规划态和开始实现时，默认都先以 `inority-slides/assets/demo/` 为基础模板，再投影成具体项目目录
- 凡是继承 `demo` 模板的实现，都必须把可见主画布当成单一 `16:9` stage；不能为了塞内容而随意长高成自由比例页面
- 目录内必须有可直接打开的入口 `index.html`
- 如果项目走构建链路，最终必须能产出 `dist/index.html`
- slides 项目必须既能脱离 handbook 独立打开，也能作为 handbook 的嵌入内容被加载
- 预览用的 lightbox 页面或预览资源页面也必须能在 handbook 项目里直接访问，不能只存在于 planning 文档链接里
- 如果某份 slides 规划最终只能导出成普通 Markdown、PDF 或 office 文件，而没有对应静态项目落点，默认视为未收敛完成

## 常见证据

- 目标受众、汇报场景、时间预算、讲述重点
- 用户对每个 section 主线的明确确认
- 用户对每个 slide 页目标的明确确认
- 来源文档、数据表、截图、图表、参考 deck、品牌模板
- 架构图源、线框草图、页面流转草图、导出的 `SVG` 成品和其命名约定
- 每页交互说明、页面状态切换、热点区域、动效触发条件
- 每页素材清单：至少要冻结标题文案、正文文案、`SVG` 图意图、`SVG` 灯箱预览链接，以及必要的配图、图标、数据、引用来源
  其中 `SVG` 灯箱预览链接不是正文里那张 planning 线框图；同时该链接要以 handbook 项目可直接访问为目标来设计
- 如果该页是架构图页，`SVG 图：` 不能只写“保留架构关系”；必须明确彩色架构图的节点、分层、连线和配色会在 planning 阶段提前生成出来供 review
- 如果该页架构图来自 `DOT`，应同时说明 `DOT -> SVG` 的生成关系；但页内评审口径以最终嵌入的 `SVG` 为准
- 对 spec 已经定义过的主题，`SVG 图：` 还应说明这张图继承的是哪一种 authority 语义：例如 northbound 入口、southbound 出口、east-west underlay、control-plane 入口、distributed placement、storage plane 或 observability module boundary
- 每页还要冻结 `当前 slide 类型`，并优先从本地展示逻辑分类中选取
- 页数上限、导出格式、演讲备注、交付截止时间

## 写作约束

- 重点是把现有素材转化为可交付 deck 的执行路径，不要退化成普通 spec 或 README
- 改用大纲式层级：二级标题是大纲视图，三级标题是 section，四级标题是 slide
- `大纲视图` 开头必须先放一张思维脑图，用来总览整套 section / slide 结构
- 不能只让用户确认一次“大纲差不多”；每个 `### section` 的主线都要有单独确认，每个 `#### slide` 的页目标也都要有单独确认
- 大纲和详细 slide 顺序必须严格一一对应
- 每个 `### section` 的第一张 `#### slide` 必须是章节标题页
- 整套 slides 的最后一张 `#### slide` 必须是 `致谢`
- 每一页的 planning 内容默认至少写清：
  - GitHub 风格的目标 Note
  - 嵌入式 `SVG` 线框图
  - 交互说明
  - 精确素材清单
  - 页级验收标准
- 对品牌型 slides，`交互说明` 不应只写“下一页 / 上一页”；应显式写清页内流转、滚动推进、导航方式和最终嵌入 / 独立打开形态
- 任何 planning 资产中的 `SVG` 文本都必须在 `16:9` 舞台安全区内收口，不能靠实现时再兜底

## 页级默认结构

每个 `#### slide` 默认应包含：

1. GitHub 风格的目标 Note
2. 嵌入式 `SVG` 线框图
3. 交互说明
4. 素材清单
5. 验收标准

其中：

- `目标 Note`：必须写成 `> [!NOTE]`，块内至少包含 `页目标` 和 `QA 链接`
- `QA 链接`：必须可点击，直接回指确认该页目标的真实问答
- `SVG` 线框图：必须直接以内联 `<svg>...</svg>` 形式嵌在该页正文里，它服务于 planning 和布局冻结
- 对架构图页，如果真正的嵌入图资产已经存在，这个线框图只需表示“图放在哪里、标题和说明围绕它怎么排”，不需要重复画架构节点本身
- `交互说明`：描述翻页、点击、悬停、滚动、展开、切换或动效触发逻辑
- `素材清单`：不能只写素材类别；至少要落到 `标题：`、`文案：`、`SVG 图：`、`SVG 灯箱预览：` 四行，其中灯箱预览必须是可点击 Markdown 链接，并且可在 handbook 中直接预览
- `当前 slide 类型`：必须写成 `展示逻辑分类/变体名`，用于后续实现时直接路由到合适模板
- 如果该页的核心证据是架构图，则正文中的线框图之外，还必须在 planning 资产里准备完整彩色 `SVG` 架构图作为 review 基准
- 该彩色架构图可以由 `DOT` 渲染而来，但最终 review 的仍然是嵌入 slide 的 `SVG`

## 章节层级约定

- `##`：大纲视图
- `###`：section
- `####`：slide

在 `## 大纲视图` 内，默认顺序是：

1. `## 思维脑图`
2. `### section`
3. `#### slide`

## QA 收敛要求

- 至少要有 `1` 条问答确认整套 slides 的总目标 / 受众 / 交付形态
- 每个 `### section` 至少要有 `1` 条问答确认该 section 的主线、作用和边界
- 每个 `#### slide` 至少要有 `1` 条问答确认该页页目标、是否保留、以及是否需要调整顺序
- 每个 `#### slide` 的目标 Note 都必须带一个能点回这条问答的 QA 链接

## 大纲与详细映射规则

- `## 大纲视图` 中出现的每个 `#### slide`，都必须在详细页列表里找到唯一对应项
- 详细页列表不允许出现大纲里没有的 slide
- 如果大纲顺序是 `A -> B -> C`，详细页顺序也必须是 `A -> B -> C`
- 调整页顺序时，必须同时更新脑图、大纲和详细页内容，不能只改其中一边
- 大纲视图里的每个 `#### slide` 都必须直接包含嵌入式 `SVG` 线框图，不允许只保留文字占位
