# Operation 规划主题补充

本文件只在 authority runbook 的主交付物是“现场操作或系统状态变化”时加载。

## 适用范围

- 主机、容器、集群、中间件、数据库、网络、发布、迁移、切流、回滚
- 需要明确执行入口主机、环境边界、变更窗口、回退动作的运行手册

## 规划重点

- 先冻结环境边界：
  - 执行入口主机
  - 目标环境
  - 变更窗口
  - 影响面
- 先确认回滚触发条件和回滚可行性，再细化正向执行路径
- 优先收敛：
  - dry-run / check / plan 是否可用
  - 是否存在只读 reconnaissance 入口
  - 是否需要 stop-the-world 或流量摘除
  - 是否需要双人复核、审批或变更单

## 常见提问维度

- `goal`
  - 最终系统状态是什么
- `risk`
  - 最高生产风险是什么，是否允许中断
- `rollback`
  - 回滚点在哪里，回滚后要恢复到什么状态
- `acceptance`
  - 哪些运行时指标、拓扑状态或服务探针构成通过证据
- `path selection`
  - 发布/迁移/切流路线是否唯一

## 执行计划偏好

- 优先按时间和环境边界拆编号项，不要按代码目录拆
- 尽量把“只读冻结”“破坏性变更”“验收观察”“回滚演练”分成独立编号项
- 如果跨多台主机，优先按主机边界或控制面边界拆项，不要把上传、远端执行、验证揉进一个大项
- 如有原生 no-op 入口，默认在前置项显式保留一次 dry-run / plan 结果

## 风险收敛提醒

- 不要把执行中需要临场判断的分叉留到正文里
- 不要把“确认是否影响业务”写成执行项；这类问题必须在规划态先收敛
- 不要把回滚方案写成一句抽象口号；至少要有触发条件、入口和完成标准
