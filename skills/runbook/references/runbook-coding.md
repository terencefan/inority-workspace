# Coding Runbook

适用于主路径围绕代码、配置、构建、测试、产物或仓库内变更推进的 runbook。

## 何时加载

- 目标的主要落地点在代码仓、配置仓、镜像构建链、CI、测试或发布产物
- 关键风险来自代码改动、构建失败、测试缺口、配置差异或发布内容不一致
- 执行路径的核心是“改什么、怎么验证、如何交付产物”

## 规划重点

- 第一个执行项必须是 `保证工作区干净`，先确认工作树、分支、未提交变更和生成产物状态，再进入后续代码变更路径
- 明确当前代码 / 配置 / 产物现状与目标版本之间的差异
- 把“代码修改”“构建 / 打包”“测试 / 验收”“交付 / 发布”拆成清晰步骤
- 先冻结基线版本、当前分支、当前配置或当前产物，再规划变更步骤
- 明确哪些步骤是只读检查，哪些是幂等生成，哪些会改写代码、配置或产物
- 优先收敛：
  - 目标分支和工作分支策略
  - 当前工作区是否干净；如果不干净，必须先向用户收敛是“先提交当前改动”还是“先 stash 当前改动”
  - 当前代码、测试、文档的基线证据
  - 最小验证闭环
  - 提交与评审单收口方式
- 把“实现”“验证”“文档同步”“提交并发起 PR/MR”视为默认交付链，而不是只停在本地改完代码

## 常见提问维度

- `goal`
  - 最终要交付的代码行为、文档和评审产物是什么
- `non-goal`
  - 哪些顺手清理或扩展能力不纳入这次改动
- `risk`
  - 最高行为回归风险、接口兼容风险、迁移风险是什么
- `acceptance`
  - 以哪些测试、构建、静态检查、手工验证作为通过标准
- `path selection`
  - 是最小修复、结构重构，还是新增抽象层

## 常见证据

- `git` 分支、commit、diff、tag
- 构建命令、测试命令、lint/typecheck 输出
- 镜像 tag、制品路径、checksum、发布包清单
- 配置文件差异、模板渲染结果、环境变量清单

## 写作约束

- 不要把“目标设计”重新写一遍；只写把代码 / 配置 / 产物从现状推进到目标所需的执行路径
- 如果涉及线上或集群现场操作，保留 `coding` 为主类型，只有当运行面步骤主导风险时再切到 `operation`
- 如果核心问题已经变成版本切换、数据迁移或状态搬运，应切到 `migration`

## 执行计划偏好

- 前两项默认固定为：
  - 同步主分支最新代码，并建立工作分支
  - 冻结当前实现
- 第一项里默认必须先检查当前工作区是否干净
  - 如果工作区不干净，不要替用户静默处理
  - 必须先向用户提问，在“先提交当前改动”和“先 stash 当前改动”之间收敛
  - 在用户拍板前，不要继续写后续实现步骤为可直接执行
- 中间项按“实现边界”拆，不要把无关模块揉进一个大项
- 验证项要显式写命令或检查入口
- 最后一项默认必须收口到提交并发起 PR/MR；如果受权限或环境限制，要写出阻塞条件和替代交付物

## 风险收敛提醒

- 不要把架构选型分歧伪装成实现步骤
- 不要把“补测试”写成泛化口号；要明确测试入口和覆盖目标
- 不要把“提交代码”当成终点；评审单或明确交接动作才是代码类 authority 的默认收口
