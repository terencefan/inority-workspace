# Operation Runbook

适用于主路径围绕服务、主机、集群、网络、权限或现场运行操作推进的 runbook。

## 何时加载

- 目标主要落在运行时现场，而不是代码仓本身
- 关键风险来自主机 / 集群状态、网络路径、权限边界、资源配置、服务启停或环境差异
- 执行路径的核心是“现场现在是什么、要操作什么、如何确认终态”

## 规划重点

- 第一个执行项必须是 `冻结现场`，先固定运行时现状、入口、权限边界和关键状态，再进入后续操作
- 先冻结现场，再决定执行入口、主机边界、权限边界和影响面
- 把上传 / 分发 / 远端执行 / 验收按 scope 拆开，不写跨主机大杂烩步骤
- 对运行面组件，`安装` 与 `启用 / restart / expose listener` 必须拆成不同编号项；不要把包安装、配置落盘、systemd 启用、服务重启、端口暴露混成同一个 stop boundary
- 把现场清理、停止条件、回滚边界写得比普通 coding runbook 更明确
- 所有运行命令都要能回答“会改哪里、影响谁、失败后卡在哪”
- 先冻结环境边界：
  - 执行入口主机
  - 目标环境
  - 变更窗口
  - 影响面
- 先确认回滚触发条件和回滚可行性，再细化正向执行路径
- 优先收敛：
  - dry-run / check / plan 是否可用
  - 是否存在只读 reconnaissance 入口
  - 是否需要 stop-the-world 或流量摘除
  - 是否需要双人复核、审批或变更单

## 常见提问维度

- `goal`
  - 最终系统状态是什么
- `risk`
  - 最高生产风险是什么，是否允许中断
- `rollback`
  - 回滚点在哪里，回滚后要恢复到什么状态
- `acceptance`
  - 哪些运行时指标、拓扑状态或服务探针构成通过证据
- `path selection`
  - 发布/迁移/切流路线是否唯一

## 常见证据

- 主机状态、服务状态、端口 / 路由 / DNS / Ingress 状态
- 配置文件、systemd、容器、Pod、Deployment、Node、Volume 状态
- 只读侦察命令输出、日志片段、监控面板、健康检查结果
- 运维平台、控制面、跳板机或集群控制面返回的只读证据

## 写作约束

- 不要把纯代码改造当成 operation runbook；如果主差异在仓库内，优先 `coding`
- 如果核心任务是旧状态向新状态切换，并伴随数据 / 流量 / 版本迁移，优先 `migration`
- 运行面步骤必须明确现场入口和权限前提，不能默认执行者自己补齐

## 执行计划偏好

- 优先按时间和环境边界拆编号项，不要按代码目录拆
- 尽量把“只读冻结”“破坏性变更”“验收观察”“回滚演练”分成独立编号项
- 如果跨多台主机，优先按主机边界或控制面边界拆项，不要把上传、远端执行、验证揉进一个大项
- 如有原生 no-op 入口，默认在前置项显式保留一次 dry-run / plan 结果

## 风险收敛提醒

- 不要把执行中需要临场判断的分叉留到正文里
- 不要把“确认是否影响业务”写成执行项；这类问题必须在规划态先收敛
- 不要把回滚方案写成一句抽象口号；至少要有触发条件、入口和完成标准
