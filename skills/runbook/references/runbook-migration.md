# Migration Runbook

适用于主路径围绕数据、状态、流量、版本、平台或控制面迁移推进的 runbook。

## 何时加载

- 目标是把现有对象从旧状态切换到新状态，而不是只做局部修改
- 关键风险来自双写、回切、兼容性、数据一致性、顺序依赖、窗口控制或 cutover
- 执行路径的核心是“迁前冻结、迁移过程、切换、迁后验证、必要时回退”

## 规划重点

- 第一个执行项必须是 `冻结现场`，先固定 source / target 当前状态、窗口和切换前基线，再进入迁移步骤
- 明确 source / target、迁移窗口、切换点、回退点和最终终态
- 把迁前准备、预演 / dry-run、正式迁移、切换、迁后校验、清理旧状态拆开
- 对每个编号项明确它是在缩小哪一条 source-target 差异
- 对不可逆步骤、双写阶段、回切边界和终止条件单独加粗说明

## 常见证据

- 源端 / 目标端版本、schema、数据量、对象数、校验和
- 同步状态、复制状态、回填进度、流量切换前后指标
- 兼容性检查、读写探针、端到端验收结果
- 切换窗口、冻结窗口、回退触发条件

## 写作约束

- 不要把 migration 写成普通 operation checklist；迁移必须显式回答 source、target、cutover、rollback
- 如果只是代码和配置改动，没有真实状态搬运或切换，不要滥用 `migration`
- 如果迁移步骤依赖代码交付或运行面操作，可以引用 `coding` / `operation` 证据，但主路径仍以迁移顺序为中心
