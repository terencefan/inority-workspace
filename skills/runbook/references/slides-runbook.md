# Slides Runbook

适用于主路径围绕演示文稿、汇报 deck、培训课件或评审 slides 产出推进的 runbook。

触发前提：用户明确要求“用 runbook 写 PPT / slide / 演示稿”。不要把普通 spec、README、总结文档或泛化写作任务误判成 `slides`。

## 何时加载

- 目标主要是交付一份 slides / PPT / Markdown slide deck，而不是直接变更运行面或代码仓状态
- 关键风险来自受众定位、叙事顺序、素材口径、图表证据、品牌模板或导出格式不一致
- 执行路径的核心是“现在有哪些素材、要讲给谁、如何收敛成可交付 deck”

## 规划重点

- 第一个执行项必须是 `冻结素材`，先固定受众、场景、来源素材、模板口径和交付格式，再进入后续制稿路径
- 从第 2 个执行项开始，每个编号项默认只对应一页 slide；一页的叙事、图表、讲述重点和验收都在该编号项内闭合
- 先冻结素材边界，再决定章节骨架、图表取舍、页面顺序和 speaker notes 范围
- 把素材盘点、故事线 / 大纲、逐页产出、统一格式、最终导出 / 交付拆成不同编号项；逐页产出阶段不要把两页合并成一个步骤
- 明确哪些步骤是只读搜集素材，哪些步骤会改写现有 deck、覆盖图表或重新导出产物
- 对评审口径、页数预算、品牌模板、最终导出格式和 handoff 责任人写清楚停止条件与验收边界

## 常见证据

- 目标受众、汇报场景、时间预算、讲述重点
- 来源文档、数据表、截图、图表、参考 deck、品牌模板
- 页数上限、导出格式、演讲备注、交付截止时间
- 审阅意见、批注清单、最终导出文件名和落点

## 写作约束

- 不要把 slides runbook 写成普通 spec 或 README；重点是把现有素材转化为可交付 deck 的执行路径
- 页级步骤标题最好直接体现页号或页名，例如 `第 3 页：现状对比`，避免执行时不知道当前编号项落在哪一页
- 如果核心工作仍是代码、配置或现场操作，只把 slides 当产物之一时，优先保持 `coding` / `operation` 为主类型
- 如果 slides 只是迁移或变更过程的汇报产物，可以引用 `migration` / `operation` 证据，但主路径仍以演示稿交付顺序为中心
